## 13. 결함 분류 및 보고 규약

### 13.1 결함 심각도 분류

| 심각도 | 정의 | 예시 | SLA |
|--------|------|------|-----|
| **Critical** | 서비스 다운, 데이터 손실, 보안 침해, 핵심 동선 차단 | 점심 시간 다운, JWT 우회, 쿠폰 위변조 | 즉시 대응 (1시간) |
| **Major** | 핵심 기능 일부 동작 불가, 우회 수단 없음 | 쿠폰 발급 실패, 매장 등록 차단, 본인 인증 실패 | 4시간 |
| **Minor** | 핵심 기능 동작 가능, 부수적 결함 | 정렬 오류, 토스트 미노출, 일부 카드 사진 누락 | 1영업일 |
| **Trivial** | UI/UX 미세 결함, 오타, 색상 불일치 | 폰트 1px 차이, 띄어쓰기 오타 | 1주일 |

### 13.2 결함 보고 양식 (GitHub Issue)

| 필드 | 내용 |
|------|------|
| 제목 | `[결함][심각도] TC ID — 한 줄 요약` (예: `[결함][Critical] TC-F3-005 — 100m 검증 우회 가능`) |
| Labels | `bug`, `severity:critical|major|minor|trivial`, `gate:m2|m3|m4`, `area:user|owner|admin|infra` |
| 본문 섹션 | 1. 환경 (스테이징 URL, 디바이스, 브라우저)<br>2. TC ID + 재현 단계<br>3. 기대 동작 vs 실제 동작<br>4. 스크린샷/로그/네트워크<br>5. 영향 범위 (사용자/사장님/어드민)<br>6. 임시 우회 방법 (있는 경우) |
| 담당자 | 자동 라우팅 — Frontend/Backend/Infra 팀별 라벨 매핑 |
